# Персональный трек обучения по пакетам курсов

## 1. Формулировка проблемы

В Университете Искусственного Интеллекта есть большое количество курсов по ИИ для разных уровней подготовки, ролей и технологических направлений. Когда слушатель покупает отдельный курс, программа обучения понятна: есть один курс, его структура, уроки, материалы и задания.

При пакетной продаже ситуация меняется. Слушатель получает доступ сразу к нескольким курсам, но не всегда понимает, с чего начать, какие материалы проходить обязательно, какие можно оставить на потом и в какой последовательности двигаться к своей цели. В результате он начинает переключаться между курсами, теряет фокус и хуже понимает ценность приобретенного пакета.

Проблема не в недостатке материалов, а в отсутствии персональной навигации по ним.

## 2. Рабочая гипотеза

Если использовать структурированный список курсов, уроков, описаний, материалов и заданий, то можно с помощью бота определить текущие потребности слушателя и сформировать для него понятный персональный трек обучения.

Бот должен не просто рекомендовать отдельные курсы, а собирать логичную последовательность прохождения: от стартовых материалов к более сложным, с учетом цели, уровня, роли, доступного времени и желаемого результата. Диагностика должна проходить в формате естественного диалога с LLM, а не в виде жесткого опроса.

## 3. Цель проекта

Создать бота-навигатора, который помогает слушателю после покупки пакета курсов получить персональный трек обучения: что проходить, в каком порядке, зачем нужен каждый этап и какой практический результат ожидается после прохождения.

Контроль прохождения, оценки, домашние задания и фиксация прогресса остаются в LMS и личном кабинете слушателя. Цель бота — не контролировать обучение, а дать понятный маршрут.

Для MVP интерфейсом может быть Telegram-бот, backend — Python/FastAPI. Такой вариант хорошо соответствует концепции: Telegram дает естественный диалог со слушателем, а FastAPI позволяет отдельно развивать синхронизацию каталога, генерацию треков и будущие интеграции.

## 4. Источник данных

В качестве основного источника используется таблица с курсами и материалами:

- каждый лист соответствует отдельному курсу или учебному блоку;
- внутри листа есть уроки, модули, названия занятий, описания, материалы, домашние задания Lite/Pro и ссылки;
- часть курсов содержит факультативные блоки, которые можно подключать как дополнительные ветки трека.

Для качественной работы бота данные нужно привести к единой структуре: курс, модуль, урок, уровень, тема, результат, prerequisites, практическая ценность, обязательность, рекомендуемый порядок.

Так как таблица может меняться, каталог должен быть динамическим: система должна регулярно синхронизировать Google Sheets, определять новые, измененные и удаленные записи, а треки строить только по актуальной версии нормализованного каталога.

## 5. Что должен делать бот

1. Вести естественный диалог со слушателем.
2. Определять цель обучения: профессия, навык, прикладная задача или проект.
3. Оценивать стартовый уровень по самооценке и уточняющим вопросам.
4. Учитывать ограничения: время в неделю, желаемая глубина, технический опыт, интересующие инструменты.
5. Подбирать релевантные курсы, модули и уроки из пакета.
6. Формировать последовательность обучения с пояснением логики.
7. Отмечать обязательные, дополнительные и факультативные материалы.
8. Показывать ожидаемый результат каждого этапа.

## 6. Что бот не должен делать

- заменять LMS;
- проверять выполнение домашних заданий;
- вести журнал прогресса;
- выставлять оценки;
- принимать решение о завершении курса;
- давать слушателю хаотичный список всех доступных материалов без приоритизации.

## 7. Пример логики персонального трека

Если слушатель хочет научиться создавать AI-агентов, но не имеет сильного технического опыта, маршрут может начинаться с базовых инструментов и no-code/vibe coding подходов, затем переходить к Telegram-ботам, памяти, базам знаний, RAG, API и продвинутому Python.

Если слушатель уже разработчик, маршрут можно сократить: убрать вводные блоки, быстрее перейти к API, backend, LangGraph, RAG, локальным моделям и интеграциям.

Если слушатель из продаж или маркетинга, маршрут должен идти через прикладные сценарии: AI-консультант, нейро-продажник, сбор данных из диалога, автоматизация воронки, no-code инструменты и только затем техническое углубление.

## 8. Минимальная версия продукта

Для MVP достаточно:

1. Нормализовать таблицу курсов в единый каталог.
2. Добавить к каждому курсу и уроку теги: уровень, роль, технология, результат, обязательность, prerequisites.
3. Подготовить сценарий диалога, который заполняет профиль слушателя без жесткого опроса.
4. Сформировать правила подбора и порядка материалов.
5. Сделать шаблон ответа бота с персональным треком.

LLM-провайдер для MVP: OpenRouter. Базовая модель задается конфигурацией, например `deepseek/deepseek-v4-flash:free`, чтобы ее можно было заменить при изменении лимитов или доступности.

Техническая основа MVP:

- Telegram-бот как интерфейс общения со слушателем;
- Python/FastAPI как backend;
- aiogram 3.x как framework для Telegram Bot API;
- OpenRouter как LLM-провайдер;
- Google Sheets как исходный каталог курсов;
- SQLite как хранилище MVP;
- нормализованный каталог как слой данных для маршрутизации.

## 9. Формат результата для слушателя

Бот должен выдавать не просто список курсов, а структурированный маршрут:

- цель трека;
- краткое объяснение выбранной логики;
- этапы прохождения;
- курсы, модули или уроки внутри каждого этапа;
- что является обязательным, а что дополнительным;
- ожидаемый практический результат;
- ориентировочная длительность при выбранной нагрузке.

## 10. Основные риски

- описания курсов могут быть слишком разнородными для автоматического подбора;
- в таблице может не хватать явных уровней сложности и входных требований;
- один и тот же материал может подходить разным аудиториям, но с разной глубиной;
- без нормализации данных бот будет давать красивые, но не всегда точные рекомендации;
- слишком длинный трек может снова создать ощущение перегруза.

## 11. Критерии успешности

Проект можно считать успешным, если слушатель после диагностики получает маршрут, в котором понятно:

- с чего начать;
- почему выбран именно такой порядок;
- какие материалы обязательны;
- какие материалы можно пройти позже;
- какой результат ожидается после каждого этапа;
- как трек связан с его личной целью.

## 12. Следующий практический этап

Следующий этап — подготовить нормализованную таблицу для MVP и механизм ее обновления. Для этого нужно взять текущую Google-таблицу с курсами и создать отдельный слой данных, где для каждого курса, модуля и урока будут заполнены поля маршрутизации: уровень, роль, цель, технологии, навыки, prerequisites, обязательность, позиция в треке, статус записи и версия синхронизации.

## 13. Развитие проекта

Следующий уровень после MVP — не только подбирать готовые курсы, но и собирать персональный курс из отдельных занятий под конкретного слушателя.

В таком режиме бот сможет:

- брать занятия из разных курсов;
- выстраивать их в новую логичную последовательность;
- исключать лишние или дублирующиеся материалы;
- добавлять факультативные ветки под развитие цели;
- обновлять маршрут, если цель слушателя изменилась;
- формировать не просто трек по пакету, а индивидуальную программу обучения.

Например, слушатель начинает с цели "сделать AI-ассистента для продаж", а позже хочет развить его до Telegram-бота с базой знаний. Тогда маршрут может расшириться: промпты → диалоговый ассистент → сбор данных → Telegram-бот → память → RAG → интеграции.
