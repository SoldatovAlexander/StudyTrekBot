# Стек проекта

## Принятое решение

Для MVP проекта принимается следующий стек:

```text
Язык: Python
Backend: FastAPI
Telegram bot framework: aiogram 3.x
LLM provider: OpenRouter
LLM model: deepseek/deepseek-v4-flash:free
Источник каталога: Google Sheets
Хранилище MVP: SQLite
Хранилище развития: PostgreSQL
```

## Почему aiogram

Для проекта нужен именно Telegram-бот, который:

- принимает сообщения от слушателя;
- ведет диалог;
- хранит состояние;
- работает через webhook;
- отправляет структурированный персональный трек.

`aiogram` хорошо подходит под эту задачу, потому что работает поверх Telegram Bot API, поддерживает асинхронную обработку, routers, middleware и FSM.

## Почему не Telethon

`Telethon` не выбран как основной инструмент для MVP, потому что он относится к другому классу задач: работа с Telegram как MTProto-клиентом, чтение каналов, история сообщений, автоматизация от имени аккаунта.

Для нашего MVP это избыточно. Бот-навигатор не должен действовать как пользовательский Telegram-клиент. Ему достаточно Telegram Bot API.

Telethon можно рассмотреть позже, если появятся задачи:

- читать Telegram-каналы как источник материалов;
- анализировать историю чатов;
- работать с Telegram-группами вне возможностей Bot API;
- автоматизировать действия от имени пользовательского аккаунта.

## Базовая архитектура

```text
Telegram user
  ↓
aiogram 3.x
  ↓
FastAPI webhook
  ↓
Dialog Manager
  ↓
OpenRouter / DeepSeek V4 Flash free
  ↓
Track Builder
  ↓
Normalized Catalog
  ↑
Google Sheets Sync
```

## Конфигурация

Модель OpenRouter не должна быть жестко зашита в коде. Все ключевые параметры задаются через переменные окружения:

```text
TELEGRAM_BOT_TOKEN=...
OPENROUTER_API_KEY=...
OPENROUTER_BASE_URL=https://openrouter.ai/api/v1
OPENROUTER_MODEL=deepseek/deepseek-v4-flash:free
GOOGLE_SHEETS_SPREADSHEET_ID=...
DATABASE_URL=...
```

## MVP-хранилище

Для MVP принимается SQLite.

Причины:

- проще запускать локально;
- не нужен отдельный database server;
- достаточно для проверки гипотезы;
- удобно хранить каталог, версии, профили и сгенерированные треки в одном файле.

PostgreSQL оставляем для развития проекта, когда появятся:

- много одновременных пользователей;
- регулярная фоновая синхронизация;
- административный интерфейс;
- интеграции;
- требования к надежности и масштабированию.

## Принятые границы

В MVP не используем:

- Telethon как основной Telegram-инструмент;
- пользовательские Telegram-аккаунты;
- frontend;
- LMS-интеграцию;
- проверку домашних заданий;
- контроль прохождения курсов.
