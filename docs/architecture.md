# Архитектура MVP

MVP должен учитывать две особенности:

- Google-таблица с курсами меняется: курсы и занятия могут добавляться, удаляться и редактироваться;
- слушатель не должен проходить жесткий опрос, взаимодействие должно выглядеть как обычный разговор с LLM.

## Общая схема

```text
Google Sheets
    ↓
Синхронизация сырого каталога
    ↓
Нормализация и обогащение данных
    ↓
Индекс материалов для поиска
    ↓
Telegram-диалог с LLM
    ↓
Персональный трек обучения
```

## Компоненты

### 1. Синхронизация Google Sheets

Система должна регулярно читать Google-таблицу и сохранять снимок данных.

Что нужно фиксировать:

- ID таблицы;
- название вкладки;
- ID вкладки;
- номер строки;
- исходные значения ячеек;
- время синхронизации;
- хеш строки или содержимого урока;
- статус записи: `active`, `changed`, `deleted`, `new`.

Такой слой позволяет не зависеть от ручного копирования и видеть, что изменилось в источнике.

### 2. Нормализованный каталог

После чтения сырой таблицы данные приводятся к единому виду:

- курс;
- модуль или блок;
- урок;
- описание;
- материалы;
- домашние задания;
- ссылки;
- уровень;
- роли;
- темы;
- технологии;
- prerequisites;
- позиция в треке.

Исходные данные не перезаписываются. Нормализованный каталог строится поверх них.

### 3. Обогащение данных

Не все поля есть в таблице явно. Например, уровень сложности, роли, prerequisites и позиция в треке могут определяться полуавтоматически через LLM.

Правильный режим для MVP:

1. LLM предлагает теги и уровень.
2. Методист или администратор проверяет спорные элементы.
3. Проверенные поля используются для генерации треков.

Важно: бот для слушателя не должен каждый раз заново анализировать всю таблицу. Он должен работать по подготовленному каталогу.

### 4. Диалоговый слой

Вместо анкеты бот ведет разговор. Внутри системы все равно есть диагностические поля, но слушателю они не показываются как форма.

Для MVP диалоговый слой реализуется через Telegram-бота. Backend — Python/FastAPI, который принимает webhook от Telegram, хранит состояние диалога и вызывает OpenRouter.

Пример внутренних полей:

```json
{
  "goal": null,
  "target_result": null,
  "technical_level": null,
  "role": null,
  "weekly_time": null,
  "depth": null,
  "known_tools": [],
  "priority_topics": [],
  "constraints": [],
  "free_goal": null
}
```

LLM должна постепенно заполнять эти поля из естественного разговора.

### 5. Генерация трека

После заполнения ключевых полей бот:

1. выбирает целевую ветку;
2. подбирает материалы из актуального каталога;
3. сортирует их по prerequisites и уровню;
4. ограничивает объем под доступное время;
5. объясняет маршрут человеческим языком;
6. явно отделяет обязательное от факультативного.

В MVP трек может строиться на уровне курсов, модулей и ключевых уроков. В развитии проекта Track Builder должен уметь собирать персональный курс из отдельных занятий разных курсов.

## Технический стек MVP

```text
Python
FastAPI
aiogram 3.x
Telegram Bot API
OpenRouter
DeepSeek V4 Flash free
Google Sheets API
SQLite для MVP-хранилища
PostgreSQL для развития
```

Рекомендуемый backend-подход:

- aiogram 3.x обрабатывает Telegram Bot API updates;
- FastAPI принимает Telegram webhook;
- отдельный сервис синхронизирует Google Sheets;
- отдельный модуль нормализует каталог;
- отдельный модуль ведет диалог;
- отдельный модуль собирает трек.

## Работа с изменениями в таблице

### Добавился курс или урок

Система помечает запись как `new`, нормализует ее и добавляет в индекс после проверки или автоматического обогащения.

### Изменился курс или урок

Система помечает запись как `changed`, сохраняет старую и новую версии. Если изменилось только описание или ссылка, маршрут можно продолжать строить автоматически. Если изменилась структура курса, нужен повторный расчет тегов.

### Удален курс или урок

Система помечает запись как `deleted`, но не удаляет сразу из истории. Новые треки не должны включать удаленные материалы.

### Изменилась вкладка

Если переименовалась вкладка, система должна сопоставлять ее по `sheetId`, а не только по названию.

## OpenRouter и модель

Для MVP используем OpenRouter как LLM-провайдера.

Рекомендуемые параметры окружения:

```text
OPENROUTER_API_KEY=...
OPENROUTER_MODEL=deepseek/deepseek-v4-flash:free
OPENROUTER_BASE_URL=https://openrouter.ai/api/v1
```

Модельный ID не должен быть жестко зашит в коде. Бесплатные модели, лимиты и доступность могут меняться, поэтому модель должна задаваться конфигурацией.

## Что важно для качества

- не отправлять в LLM всю таблицу целиком при каждом диалоге;
- использовать актуальный нормализованный каталог;
- хранить версию каталога, по которой был построен трек;
- показывать слушателю только понятный маршрут, а не технические теги;
- при нехватке данных задавать 1-2 уточняющих вопроса, а не запускать длинную анкету.

## Развитие до персонального курса

После проверки MVP проект можно расширить: бот будет не только выбирать готовые курсы, но и собирать индивидуальную программу из отдельных занятий.

Для этого потребуется:

- маршрутизация на уровне уроков;
- явные prerequisites между занятиями;
- оценка дублирования тем;
- группировка занятий в новые этапы;
- сохранение версии персонального курса;
- возможность пересобрать курс при изменении цели слушателя.
